# runBlocking: blessed uses and per-situation alternatives

Context: [itsramiel/Software-Engineering#2](https://github.com/itsramiel/Software-Engineering/issues/2). Findings feed `docs/development/kotlin-rules.md`.

All claims below are sourced from kotlinx.coroutines API docs, the Kotlin coroutines guide, kotlinx-coroutines-test docs, Android developer docs, and kotlinx.coroutines maintainer statements (Roman Elizarov / elizarov, Vsevolod Tolstopyatov / qwwdfsad, Dmitry Khalanski / dkhalanskyjb) in the project's issue tracker. A notable negative finding: Elizarov's widely-cited Medium posts ([Blocking threads, suspending coroutines](https://elizarov.medium.com/blocking-threads-suspending-coroutines-d33e11bf4761), [Structured concurrency](https://elizarov.medium.com/structured-concurrency-722d765aa952)) never mention `runBlocking` by name — they establish the convention framework ("suspending functions do not block the caller thread"; blocking work belongs behind `withContext(Dispatchers.Default/IO)`). The `runBlocking`-specific positions live in the API docs, the guide, and GitHub issues.

## 1. What runBlocking is, per its own contract

The [runBlocking API doc](https://kotlinlang.org/api/kotlinx.coroutines/kotlinx-coroutines-core/kotlinx.coroutines/run-blocking.html):

> "Runs the given block in-place in a new coroutine based on context, blocking the current thread until its completion, and then returning its result."
>
> "It is designed to bridge regular blocking code to libraries that are written in suspending style, to be used in `main` functions, in tests, and in non-`suspend` callbacks when `suspend` functions need to be called."

Two contract details matter for the failure modes below (same doc):

- "By default, an event loop opened on this thread is used as the ContinuationInterceptor." That event loop "is shared with the other `runBlocking` calls. This can lead to surprising and undesired behavior" — nested `runBlocking` calls "may execute other coroutines' tasks from the same event loop instead of running their own tasks".
- On the JVM, interrupting the blocked thread cancels the coroutine job and makes `runBlocking` throw `InterruptedException`.

`runBlocking` exists only where a thread can be blocked: JVM and Native. It does not exist on Kotlin/JS ([Kotlin/kotlinx.coroutines#1180](https://github.com/Kotlin/kotlinx.coroutines/issues/1180); correspondingly, [runTest](https://kotlinlang.org/api/kotlinx.coroutines/kotlinx-coroutines-test/kotlinx.coroutines.test/run-test.html) "on JS ... creates a `Promise`" instead of blocking).

## 2. Blessed uses

**`main` entry points, when the entry point cannot be `suspend`.** The [coroutines guide](https://kotlinlang.org/docs/coroutines-basics.html): "While you can mark the `main()` function as `suspend` in some projects, it may not be possible when integrating with existing code or using a framework. In that case, check the framework's documentation to see if it supports calling suspending functions. If not, use `runBlocking()` to call them by blocking the current thread." Note the ordering: `suspend fun main` is the first choice; `runBlocking` is the fallback.

**Implementing a blocking interface/callback that must call suspend code.** The guide's rule is scoped exactly this narrowly: "Use `runBlocking()` only when there is no other option to call suspending code from non-suspending code" ([coroutines guide](https://kotlinlang.org/docs/coroutines-basics.html)), with a non-`suspend` repository interface as the example. Maintainer dkhalanskyjb states the same: "`runBlocking` is useful fairly situationally, typically for implementing existing interfaces that expect their members to be normal blocking functions" ([Kotlin/kotlinx.coroutines#3456](https://github.com/Kotlin/kotlinx.coroutines/issues/3456)). This is the "bridge to blocking/Java callers" case from the API doc's design statement.

**Tests — historically.** The API doc lists tests among the designed uses, but for coroutine code this role has been taken over by `runTest` (section 4). The residual legitimate test uses are narrow.

**Deliberate, eyes-open blocking — including, rarely, on a UI thread.** This is the most surprising maintainer position on record. Elizarov proposed making `runBlocking` throw on the Android UI thread ("using `runBlocking` from Android UI thread shall throw an exception to fail-fast and prevent hard-to-diagnose bugs when the code just 'happens to run fast' during testing but can in fact hang the UI in production" — [#227](https://github.com/Kotlin/kotlinx.coroutines/issues/227)). The check shipped in 0.24.0 (`IllegalStateException: runBlocking is not allowed in Android main looper thread`) — and was **reverted in 0.25.0** ([#464](https://github.com/Kotlin/kotlinx.coroutines/issues/464), elizarov: "It is released in 0.25.0. Limitation on `runBlocking` are reverted."). qwwdfsad's revert rationale ([#464](https://github.com/Kotlin/kotlinx.coroutines/issues/464)): "There are cases when blocking in UI is legit" (users cited app initialization in `Application#onCreate` that must complete before startup proceeds), the check broke UI-thread test frameworks, and the opt-out property risked tests passing where production would crash. The same maintainer had written "In general, blocking UI thread is a no go, so we prefer to have this check enabled by default" ([#227](https://github.com/Kotlin/kotlinx.coroutines/issues/227)). Net position: blocking the UI thread is presumed wrong, but the library will not forbid it because rare deliberate uses exist — a rule for humans, not a mechanical invariant.

## 3. Failure modes

**Thread starvation: runBlocking inside a coroutine / suspend function.** The [API doc](https://kotlinlang.org/api/kotlinx.coroutines/kotlinx-coroutines-core/kotlinx.coroutines/run-blocking.html) singles this out: "Calling `runBlocking` from a suspend function is redundant" — instead of releasing the thread when the inner code suspends, "it will block, potentially leading to thread starvation issues." (Earlier revisions of the same doc said flatly "This function should not be used from a coroutine", quoted by dkhalanskyjb in [#3456](https://github.com/Kotlin/kotlinx.coroutines/issues/3456).) The starvation is concrete because dispatcher pools are finite: [Dispatchers.IO](https://kotlinlang.org/api/kotlinx.coroutines/kotlinx-coroutines-core/kotlinx.coroutines/-dispatchers/-i-o.html) "defaults to the limit of 64 threads or the number of cores (whichever is larger)", and IO "and its views share threads with the Default dispatcher". Every thread parked in `runBlocking` is deducted from the pool that must eventually run the work it is waiting on — at the limit, that is a deadlock, not just slowness.

**Deadlock: nested runBlocking / runBlocking on an event-loop thread.** Elizarov documented the minimal reproducer in [#860](https://github.com/Kotlin/kotlinx.coroutines/issues/860): code that "uses `runBlocking { ... }` to provide integration for blocking Java APIs ... can deadlock when being run inside another `runBlocking { ... }`" — the inner call blocks the only thread the outer event loop has (the real-world origin was Ktor, ktorio/ktor#672). The [API doc](https://kotlinlang.org/api/kotlinx.coroutines/kotlinx-coroutines-core/kotlinx.coroutines/run-blocking.html) generalizes: the shared per-thread event loop "can lead to surprising and undesired behavior", including tasks of other coroutines executing inside your `runBlocking` and potential stack overflow. The structural point: a blocking bridge buried inside a library is a latent deadlock for any caller that is itself inside a coroutine — which is why the bridge belongs only at the outermost boundary.

**ANR / frozen UI: runBlocking on the Android main thread.** Android docs: blocking the main thread means "the OS isn't able to call `onDraw()`, which causes your app to freeze and potentially leads to an Application Not Responding (ANR) dialog" ([Coroutines on Android](https://developer.android.com/kotlin/coroutines)). The [ANR doc](https://developer.android.com/topic/performance/vitals/anr) sets the budget: an ANR triggers when the app "has not responded to an input event (such as key press or screen touch) within 5 seconds" (with similar few-second limits for services and broadcast receivers). Elizarov's fail-fast rationale in [#227](https://github.com/Kotlin/kotlinx.coroutines/issues/227) names the insidious part: the code "just 'happens to run fast' during testing but can in fact hang the UI in production" — `runBlocking`'s latency is the latency of whatever it awaits, so main-thread uses pass review and CI, then ANR in the field on a slow network.

**Hidden blocking behind a suspend signature.** The convention that makes coroutines composable is that "suspending functions do not block the caller thread" ([Elizarov, Blocking threads, suspending coroutines](https://elizarov.medium.com/blocking-threads-suspending-coroutines-d33e11bf4761)); Android states the same as a rule: "Suspend functions should be main-safe, meaning they're safe to call from the main thread" ([best practices](https://developer.android.com/kotlin/coroutines/coroutines-best-practices)). A suspend function that internally uses `runBlocking` (or a blocking function silently wrapping `runBlocking`) violates the contract every caller relies on — this is the root of the "works here, deadlocks there" class of bugs, and also breaks tests (section 4).

## 4. Does runTest fully supersede runBlocking in tests?

For unit tests of coroutine code: effectively yes. For tests that must interact with real time or exercise a blocking facade: no — and those are the only residual uses.

- The [kotlinx-coroutines-test README](https://github.com/Kotlin/kotlinx.coroutines/blob/master/kotlinx-coroutines-test/README.md) opens with "runTest is the way to test code that involves coroutines." [runTest](https://kotlinlang.org/api/kotlinx.coroutines/kotlinx-coroutines-test/kotlinx.coroutines.test/run-test.html) "behaves similarly to `runBlocking`, with the difference that the code that it runs will skip delays" (virtual time), plus a built-in 60-second timeout that cancels hung test bodies. Android's guidance matches: "Use `runTest` for tests that use coroutines" ([Testing Kotlin coroutines on Android](https://developer.android.com/kotlin/coroutines/test)).
- The library's own [migration guide](https://github.com/Kotlin/kotlinx.coroutines/blob/1.6.4/kotlinx-coroutines-test/MIGRATION.md) has a section literally titled "Replace `runBlocking` with `runTest`": "Now that `runTest` works properly with asynchronous completions, `runBlocking` is only occasionally useful. As is, most uses of `runBlocking` in tests come from the need to interact with dispatchers that execute on other threads, like `Dispatchers.IO` or `Dispatchers.Default`." (Before 1.6, `runBlocking` in tests was often a workaround for `runBlockingTest`'s inability to await other dispatchers; that reason is gone.)
- What runTest does **not** cover, per its own docs: virtual time does not propagate into real dispatchers — "Delays in code that runs inside dispatchers that don't use a TestCoroutineScheduler don't get skipped" ([runTest doc](https://kotlinlang.org/api/kotlinx.coroutines/kotlinx-coroutines-test/kotlinx.coroutines.test/run-test.html)); the fix is injecting `TestDispatcher`s rather than falling back to `runBlocking` ([Android testing guide](https://developer.android.com/kotlin/coroutines/test): "In tests, replace real dispatchers with instances of `TestDispatchers`"). `Dispatchers.Main` must be replaced via `Dispatchers.setMain` in local JVM tests (see section 6).
- Two hard boundaries, from maintainer dkhalanskyjb in [#3456](https://github.com/Kotlin/kotlinx.coroutines/issues/3456): calling `runBlocking` *inside* `runTest` is "completely unsupported due to the nature of `runBlocking` itself" (runTest's body is a coroutine, and the runBlocking doc forbids use from a coroutine — the inner delays run in real time and the test scheduler cannot see them). And a blocking function that wraps `runBlocking` internally cannot be tested with delay skipping at all — "it's more reliable to test the coroutine that is wrapped in `runBlocking`", i.e. test the `internal` suspend function behind the facade.
- Residual legitimate `runBlocking` in tests: a test that *wants* real wall-clock time (timing/benchmark-style tests, integration tests against real IO) — `runTest`'s entire purpose is to remove real time, so it is the wrong tool there ([runTest doc](https://kotlinlang.org/api/kotlinx.coroutines/kotlinx-coroutines-test/kotlinx.coroutines.test/run-test.html) on delay skipping being the defining difference).

**Rule shape for the rules doc:** default to `runTest` for anything coroutine-shaped; `runBlocking` in a test is acceptable only with a stated reason (real time semantics or exercising a blocking facade end-to-end), never inside `runTest`.

## 5. Misuse → concrete alternative, per situation

### Inside a suspend function (any platform)

Misuse: `runBlocking { fetchX() }` in a `suspend fun`. The [API doc](https://kotlinlang.org/api/kotlinx.coroutines/kotlinx-coroutines-core/kotlinx.coroutines/run-blocking.html) marks this "redundant" and starvation-prone. Alternative: call the suspend function directly; if a scope for parallel decomposition is needed, `coroutineScope { }` — the suspending counterpart that waits without blocking ([coroutines guide](https://kotlinlang.org/docs/coroutines-basics.html)).

### In UI code to extract a value from suspend code (Android)

Misuse: `val x = runBlocking { repo.load() }` in an Activity/ViewModel to "make it synchronous". Freezes rendering; ANR at 5 s ([ANR doc](https://developer.android.com/topic/performance/vitals/anr)). Alternative: launch instead of block — `viewModelScope.launch { }` / `lifecycleScope.launch { }` and deliver the result via state, with the scope tied to the lifecycle so work is auto-cancelled ([Coroutines on Android](https://developer.android.com/kotlin/coroutines)). The suspend functions being called must themselves be main-safe ([best practices](https://developer.android.com/kotlin/coroutines/coroutines-best-practices)).

### Wrapping blocking IO to "make it a coroutine"

Misuse: `runBlocking(Dispatchers.IO) { blockingCall() }` or exposing blocking work as suspend without moving it off-thread. This is the bridge pointed the wrong way. Alternative: `withContext(Dispatchers.IO) { blockingCall() }` inside a suspend function — the class "is in charge of moving the execution off the main thread using `withContext`" ([best practices](https://developer.android.com/kotlin/coroutines/coroutines-best-practices); [Elizarov](https://elizarov.medium.com/blocking-threads-suspending-coroutines-d33e11bf4761) for the underlying convention). If the blocking API responds to `Thread.interrupt`, wrap with [runInterruptible](https://kotlinlang.org/api/kotlinx.coroutines/kotlinx-coroutines-core/kotlinx.coroutines/run-interruptible.html)`(Dispatchers.IO) { }` to also get cancellation: it "enables single-call conversion of interruptible Java methods into suspending functions".

### Exposing coroutine-based code to Java / non-coroutine callers

If the caller can consume a future: [future](https://kotlinlang.org/api/kotlinx.coroutines/kotlinx-coroutines-core/kotlinx.coroutines.future/future.html)` { }` launches a child coroutine "without blocking the current thread" and returns a `CompletableFuture` ("equivalent to `async` in terms of structured concurrency"). If the external contract is irreducibly blocking (a legacy interface method), `runBlocking` at that boundary is the designed use ([API doc](https://kotlinlang.org/api/kotlinx.coroutines/kotlinx-coroutines-core/kotlinx.coroutines/run-blocking.html); dkhalanskyjb in [#3456](https://github.com/Kotlin/kotlinx.coroutines/issues/3456)) — but it must sit at the outermost layer, on a thread that is never a coroutine dispatcher's or another `runBlocking`'s thread, or the [#860](https://github.com/Kotlin/kotlinx.coroutines/issues/860) deadlock applies.

### `main()` / CLI entry points (JVM)

Prefer `suspend fun main`; use `runBlocking` when a framework dictates a non-suspend signature ([coroutines guide](https://kotlinlang.org/docs/coroutines-basics.html)). Both are fine — this is the blessed case.

### In tests

Misuse: `runBlocking` as the default test wrapper (real delays, no virtual-time control), or `runBlocking` inside `runTest`. Alternative: `runTest` + injected `TestDispatcher`s + `Dispatchers.setMain` for main-referencing code (section 4 sources). Keep `runBlocking` only for deliberately-real-time tests.

### Synchronous requirement at process startup (Android `Application#onCreate` and similar)

The one main-thread case maintainers explicitly declined to outlaw ([#464](https://github.com/Kotlin/kotlinx.coroutines/issues/464)). First try to remove the synchronous requirement (defer the dependent work into a launched coroutine); if the design genuinely requires the value before startup proceeds and the wait is short, a `runBlocking` here is a documented, deliberate exception — annotate it as such. The rule should demand justification, not forbid.

### Inside server request handling on coroutine-aware frameworks (Ktor, WebFlux, etc.)

Misuse: `runBlocking` inside a handler that already runs in a coroutine or on an event-loop thread — this is exactly the Ktor-originated deadlock of [#860](https://github.com/Kotlin/kotlinx.coroutines/issues/860) and the starvation case of the [API doc](https://kotlinlang.org/api/kotlinx.coroutines/kotlinx-coroutines-core/kotlinx.coroutines/run-blocking.html). Alternative: stay suspending end-to-end; bridge blocking dependencies with `withContext(Dispatchers.IO)` / `runInterruptible`, not the reverse. In thread-per-request servlet stacks (no coroutine above you), `runBlocking` at the handler boundary is the legitimate bridge case.

## 6. Platform-context differences (input for the rules-doc structure question)

The mechanism is identical everywhere — a parked thread — but what that thread is worth differs, so the verdicts differ. The rules doc should split by platform.

**Android main thread.** The blocked thread is the one rendering the UI and receiving input; cost is user-visible freeze and OS-enforced ANR at ~5 s ([ANR doc](https://developer.android.com/topic/performance/vitals/anr), [Coroutines on Android](https://developer.android.com/kotlin/coroutines)). Verdict: near-absolute "don't", with the maintainer-acknowledged startup exception ([#464](https://github.com/Kotlin/kotlinx.coroutines/issues/464)). Latency is environment-dependent, so testing does not catch it ([#227](https://github.com/Kotlin/kotlinx.coroutines/issues/227)).

**JVM server / CLI.** Blocking a dedicated thread (main, a servlet worker) is the designed use; the danger reappears when the blocked thread belongs to a shared pool or event loop — dispatcher starvation against finite pools ([Dispatchers.IO doc](https://kotlinlang.org/api/kotlinx.coroutines/kotlinx-coroutines-core/kotlinx.coroutines/-dispatchers/-i-o.html): 64-thread default, shared with Default) and nested-event-loop deadlock ([#860](https://github.com/Kotlin/kotlinx.coroutines/issues/860)). Verdict: allowed at the outermost blocking boundary only; the useful rule is about *which thread* is being blocked, not the call itself.

**Tests.** Local JVM unit tests have no Android main looper — code touching `Dispatchers.Main` throws unless `Dispatchers.setMain(testDispatcher)` is installed; do not replace Main in instrumented tests where the real UI thread exists ([Android testing guide](https://developer.android.com/kotlin/coroutines/test)). Default wrapper is `runTest`, not `runBlocking` ([README](https://github.com/Kotlin/kotlinx.coroutines/blob/master/kotlinx-coroutines-test/README.md), [migration guide](https://github.com/Kotlin/kotlinx.coroutines/blob/1.6.4/kotlinx-coroutines-test/MIGRATION.md)).

**Kotlin/JS (and multiplatform common code).** `runBlocking` does not exist — there is no thread to block ([#1180](https://github.com/Kotlin/kotlinx.coroutines/issues/1180)); `runTest` papers over this for tests by returning a `Promise`-backed `TestResult` on JS ([runTest doc](https://kotlinlang.org/api/kotlinx.coroutines/kotlinx-coroutines-test/kotlinx.coroutines.test/run-test.html)). Any common-code rule can therefore state that `runBlocking` is JVM/Native-only bridging machinery, never a portable pattern.
